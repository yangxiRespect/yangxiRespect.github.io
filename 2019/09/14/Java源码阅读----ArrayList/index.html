ArrayList and CopyOnWriteArrayList

ArrayList和CopyOnWriteArrayList是jdk中提供的列表的数组实现。ArrayList不支持并发操作，CopyOnWriteArrayList支持并发操作。

ArrayList源码

原理和目的

使用对象数组来保存数据。最大容量为Int的最大值，默认容量为10,每次扩容为原来大小的1.5倍。

  • 继承了 AbstractList类（java提供的一个抽象类，实现了List接口的部分方法）
  • 实现了 List, RandomAccess, Cloneable, java.io.Serializable等接口

其中RandomAccess接口表示ArrayList提供了随机访问列表的功能。

构造方法

ArrayList提供了3种构造方法来初始化ArrayList对象。

  • 方法一：指定初始化列表大小

该方法用于构造一个指定大小的ArrayList对象。当初始化大小小于0时，抛出IllegalArgumentException，
当初始化大小为0时，ArrayList中的对象数组初始值为EMPTY_ELEMENTDATA，区别于默认构造函数中的DEFAULTCAPACITY_EMPTY_ELEMENTDATA。

  • 方法二：空参数构造函数
    该方法用于构造默认的ArrayList对象。其初始化为10，ArrayList中的对象数组初始值为DEFAULTCAPACITY_EMPTY_ELEMENTDATA。
  • 方法三：Collection对象参数构造函数
    该方法用于将其他Collection对象转换为ArrayList对象，使用了Arrays.copyOf(elementData, size, Object[].class)方法将对象数组拷贝到ArrayList对象中。

ps：

  1. EMPTY_ELEMENTDATA于DEFAULTCAPACITY_EMPTY_ELEMENTDATA的区别，在ArrayList代码中无构造函数创建的对象数组和初始化大小为0构造的对象数组虽然都代表的是空数组。但在第一次数据添加时，决定了初始化的数组大小。在calculateCapacity方法中，进行了判断，如果数组为DEFAULTCAPACITY_EMPTY_ELEMENTDATA，并且初次添加的数量小于默认数量10时，该对象的默认大小就为10。其余情况下都返回当前minCapacity（当前大小+添加的数据大小）。

常用方法

ArrayList中的大部分数据操作方法，都在List接口中定义，包括add，get，remove等

  • add与addAll方法

提供了2个add方法，用来添加单个对象，默认在最后的位置添加，或者在指定位置添加，余下部分后移。
2个方法都调用ensureCapacityInternal进行大小判断，来决定是否进行扩容，判断完成后来，将对象放入指定位置，将size++。
在指定位置添加对象时会调用rangeCheckForAdd方法，检查index是否在范围中，如果超过范围，抛出IndexOutOfBoundsException异常。

addAll方法与add方法类似，将Collection里加入该集合。与add方法不同的是，在扩容判断完成后会调用System.arraycopy()方法，将数据加入该集合。当在指定位置添加时，会先将该索引之后的数据统一向后移，再将新数据导入。

  • get与set方法
    2个方法需要传入index参数，get方法返回当前索引位置的参数，set方法替换当前位置的参数，并且返回被替换的参数。2个方法都调用了rangeCheck(index)方法来对index进行检查，抛出IndexOutOfBoundsException异常。
  • remove与removeAll

提供了2个remove函数来输出目标对象。
根据索引来删除目标位置上的对象，返回被删除对象。在调用rangeCheck方法判断index是否合法后，取出该位置上的对象，调用System.arraycopy获取新数组，返回被删除的对象。
根据给定的对象来输出数组中第一个出现的相同对象，返回boolean值来标识是否删除成功。根据传入的参数是否为空，进行了不同的判定，但步骤相同。for循环数组，进行equals判定，如果相同调用fastRemove(index)方法，删除该对象。

ps

  1. System.arraycopy（）方法是Java提供的数组复制方法，从指定的源数组的指定位置复制数组，到目标数组的指定位置。

src 表示指定的源数组，srcPos表示源数组的指定位置，dest表示目标数组，destPos表示目标数组的指定位置，length表示复制的长度。

方法会抛出IndexOutOfBoundsException、ArrayStoreException和NullPointerException3中异常

  1. rangeCheckForAdd与rangeCheck，2个方法都用来检查index是否越界。

重要方法

  • 扩容机制
    在每次进行List的添加操作时，会计算出添加后的元素个数minCapacity，通过ensureExplicitCapacity方法，来进行是否扩容的判定，如果minCapacity大于当前数组的大小，调用grow方法进行扩容，扩容机制为当前大小的1.5倍。得到最终的大小后，调用Arrays.copyOf(elementData, newCapacity)，方法将当前数组的值导入到目标大小的新数组中。其中在newCapacity的计算中比较关键的2部判定包括（newCapacity - minCapacity < 0）和（newCapacity - MAX_ARRAY_SIZE）。在ArrayList中默认的最大元素数为private static final int MAX_ARRAY_SIZE = Integer.MAX_VALUE - 8;-8是由于有些VM会在数组中保留一些标题字。如果计算出来的newCapacity为负值，则OutOfMemoryError异常，大于MAX_ARRAY_SIZE则将最大改为Integer.MAX_VALUE。

fail-fast机制

ArrayList是非线性安全，此类的 iterator 和 listIterator 方法返回的迭代器是快速失败的：在创建迭代器之后，除非通过迭代器自身的 remove 或 add 方法从结构上对列表进行修改，否则在任何时间以任何方式对列表进行修改，迭代器都会抛出 ConcurrentModificationException。即在一方在便利列表，而另一方在修改列表时，会报ConcurrentModificationException错误。而这不是唯一的并发时容易发生的错误，在多线程进行插入操作时，由于没有进行同步操作，容易丢失数据。

CopyOnWriteArrayList源码

原理和目的

java中的列表实现与区别

java中列表实现有ArrayList、Vector、CopyOnWriteArrayList、Collections.synchronizedList(list)四种方式。
ArrayList在设计时就不支持并发修改，而Vector虽然是一个线程安全的列表，采用数组实现，但其线程安全的实现方式是对所有操作都加上了synchronized关键字，这种方式严重影响效率，因此，不再推荐使用Vector。
CopyOnWriteArrayList和Collections.synchronizedList是实现线程安全的列表的两种方式。两种实现方式分别针对不同情况有不同的性能表现，其中CopyOnWriteArrayList的写操作性能较差，而多线程的读操作性能较好。而Collections.synchronizedList的写操作性能比CopyOnWriteArrayList在多线程操作的情况下要好很多，而读操作因为是采用了synchronized关键字的方式，其读操作性能并不如CopyOnWriteArrayList。因此在不同的应用场景下，应该选择不同的多线程安全实现类。

CopyOnWrite

Copy-On-Write简称COW，是一种用于程序设计中的优化策略。其基本思路是，从一开始大家都在共享同一个内容，当某个人想要修改这个内容的时候，才会真正把内容Copy出去形成一个新的内容然后再改，这是一种延时懒惰策略。除了CopyOnWriteArrayList外，从JDK1.5开始Java并发包里还有CopyOnWriteArraySet也是使用CopyOnWrite机制实现的并发容器。CopyOnWrite容器非常有用，可以在非常多的并发场景中使用到。

什么是CopyOnWrite容器

CopyOnWrite容器即写时复制的容器。通俗的理解是当我们往一个容器添加元素的时候，不直接往当前容器添加，而是先将当前容器进行Copy，复制出一个新的容器，然后新的容器里添加元素，添加完元素之后，再将原容器的引用指向新的容器。这样做的好处是我们可以对CopyOnWrite容器进行并发的读，而不需要加锁，因为当前容器不会添加任何元素。所以CopyOnWrite容器也是一种读写分离的思想，读和写不同的容器。

具体实现

CopyOnWriteArrayList与ArrayList有着相同的方法，具体的区别在与使用CopyOnWriteArrayList添加元素时，将添加的元素加入新复制的对象数组中，再替换旧数组。该操作如果不加锁多线程写的时候会Copy出N个副本出来。

而在读的时候不需要加锁，如果读的时候有多个线程正在向CopyOnWriteArrayList添加数据，读还是会读到旧的数据，因为写的时候不会锁住旧的CopyOnWriteArrayList。所以使用CopyOnWriteArrayList只会保证数据的最终一致性。

优缺点

CopyOnWrite的优点

线程安全。CopyOnWrite是线程安全的，我们可以安全地迭代列表，即使正在进行并发修改。

CopyOnWrite的缺点

CopyOnWrite容器有很多优点，但是同时也存在两个问题，即内存占用问题和数据一致性问题。所以在开发的时候需要注意一下。

内存占用问题。因为CopyOnWrite的写时复制机制，所以在进行写操作的时候，内存里会同时驻扎两个对象的内存，旧的对象和新写入的对象（注意:在复制的时候只是复制容器里的引用，只是在写的时候会创建新对象添加到新容器里，而旧容器的对象还在使用，所以有两份对象内存）。如果这些对象占用的内存比较大，比如说200M左右，那么再写入100M数据进去，内存就会占用300M，那么这个时候很有可能造成频繁的Yong GC和Full GC。

针对内存占用问题，可以通过压缩容器中的元素的方法来减少大对象的内存消耗，比如，如果元素全是10进制的数字，可以考虑把它压缩成36进制或64进制。或者不使用CopyOnWrite容器，而使用其他的并发容器，如ConcurrentHashMap。

数据一致性问题。CopyOnWrite容器只能保证数据的最终一致性，不能保证数据的实时一致性。所以如果你希望写入的的数据，马上能读到，请不要使用CopyOnWrite容器。

总结

CopyOnWrite并发容器用于读多写少的并发场景。

  1. 减少扩容开销。根据实际需要，初始化CopyOnWriteMap的大小，避免写时CopyOnWriteMap扩容的开销。

  2. 使用批量添加。因为每次添加，容器每次都会进行复制，所以减少添加次数，可以减少容器的复制次数。如使用上面代码里的addBlackList方法。
